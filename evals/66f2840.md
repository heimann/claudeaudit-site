## Findings

### What's stable

- Maturity level: 100% stable across all runs and all fixtures. Never flipped.
- 30 of 36 category-fixture pairs (83%) were perfectly stable across 5 runs.
- "Clearly bad" fixture (manim): zero variance on any category. When signals are unambiguous, the skill is deterministic.

### What drifts

6 category-fixture pairs showed drift, all exactly 1 point (never 2+):

| Fixture | Category | Scores | Issue |
|---------|----------|--------|-------|
| fzf | Rules/conventions | 0,1,0,1,0 | BUILD.md + .editorconfig sometimes count as "basic conventions mentioned" (1) vs "no documented conventions" (0) |
| fzf | Sandbox compat | 2,1,1,2,1 | Dockerfile presence is weighed differently - sometimes "runs in Docker" (2) vs "tests need tmux which sandboxes lack" (1) |
| vscode-icons | Agent docs | 3,3,3,2,2 | CLAUDE.md @import to AGENTS.md sometimes counts as "@imports so docs stay current" (3) vs "no subdirectory CLAUDE.md files" (2) |
| vscode-icons | Permissions | 3,3,2,2,2 | .claude/settings.json with skills sometimes counts as "thoughtfully configured per-context" (3) vs "scoped but no deny list" (2) |
| glances | Dep bootstrap | 3,2,3,2,3 | Two-step make install-uv + make venv-dev sometimes counts as "fully automated" (3) vs "no single command" (2) |
| glances | Self-verification | 3,2,3,2,2 | Extensive test suite sometimes counts as "full E2E coverage" (3) vs "no quiet mode, CI only runs core" (2) |

### Pattern

Drift occurs at 2/3 and 0/1 boundaries where rubric criteria are genuinely ambiguous. The skill is interpreting the same signals differently across runs because the rubric leaves room for judgment. This is fixable by tightening the criteria for these specific boundary cases.

### Recommendations

1. For the weekly index: take the median of 3 runs to smooth out noise.
2. Tighten rubric language for the 6 drifty categories - add more concrete thresholds.
3. The "clearly bad" and "clearly good" ends of the spectrum are reliable. Focus rubric improvements on the gray areas.

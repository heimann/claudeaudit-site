# Eval Results - 5a6e905 (hybrid: haiku explore + opus score)

2026-04-01 | Explore: claude-haiku-4-5 | Score: claude-opus-4-6 | 5 runs per fixture

## Method

Two-phase hybrid approach:
1. **Explore (haiku)**: 4 parallel haiku agents per repo gather signal data (docs/structure, build/test, quality/conventions, scalability). Run once per repo - exploration is near-deterministic.
2. **Score (opus)**: Opus agent reads the frozen signal summary + rubric and produces scores/report. Run 5 times per repo with identical input to isolate scoring variance.

This separates exploration variance from judgment variance. Each opus scorer reads SKILL.md (1 tool call) and scores from the signal text - no file exploration.

### Why it works

When opus explores a repo itself, it makes different exploration choices each run
(which files to read, how deeply). These different observations produce different
evidence, which leads to different scores at boundary cases. By freezing the
evidence (haiku explores once, produces a canonical signal summary), the scorer
sees identical input every time. Scoring variance drops to near zero.

The one remaining drift (glances repo structure) is pure judgment variance: the
signal data says "plugin/export auto-discovery enforces boundaries by convention"
and opus sometimes scores that as "enforced" (3) and sometimes as "convention
only" (2). This is a rubric precision issue, not an exploration issue.

### Score differences from pure opus v2

The hybrid produced different scores for some categories because haiku's
systematic exploration surfaced signals that opus sometimes missed:

| Fixture | Category | Pure v2 | Hybrid | Why |
|---------|----------|---------|--------|-----|
| manim | Repo structure | 1 | 2 | Haiku reported clearer subdirectory structure evidence |
| fzf | Dep bootstrap | 2 | 1 | Haiku surfaced multiple undocumented system deps (tmux, zsh, fish, shfmt) |
| fzf | Rules/conventions | 0 | 1 | Haiku surfaced .editorconfig + .rubocop.yml + BUILD.md as config-level conventions |
| fzf | Level | Ok | Bad | Consequence of dep bootstrap drop (Runnable needs all >= 2) |
| glances | Worktree | 2 | 1 | Haiku surfaced host-networking + no-env-var port detail |
| glances | Sandbox | 2 | 1 | Haiku surfaced host PID + host network docker-compose requirement |
| glances | Level | Great | Good | Consequence of worktree/sandbox drop |

These are accuracy improvements, not regressions. Haiku's thorough exploration
produced more complete signal data, leading to more defensible scores.

## Cost analysis

Per-repo cost comparison (approximate):

| Approach | Explore | Score | Total agents | Opus tokens |
|----------|---------|-------|-------------|-------------|
| Pure opus | 1 opus | - | 1 | ~35k |
| Hybrid | 4 haiku | 1 opus | 5 | ~16k opus + ~50k haiku |

Haiku tokens are ~10x cheaper than opus tokens. The hybrid uses ~50% less opus
budget per audit while producing more stable and more accurate results.

For stability runs (5x per repo):
- Pure opus: 5 opus agents, ~175k opus tokens
- Hybrid: 4 haiku + 5 opus, ~80k opus + ~50k haiku tokens

## Recommendations

1. **Adopt the hybrid approach for production audits.** More stable, more accurate, cheaper.
2. **Compute maturity level programmatically** rather than leaving it to the scorer. The formula is mechanical (highest level where all categories >= 2) but scorers sometimes apply it inconsistently.
3. **For the weekly index**: hybrid with single run is sufficient (97% category stability). Only take median of 3 if budget allows.
4. **Remaining rubric work**: tighten repo structure 2/3 boundary for convention-enforced vs mechanically-enforced module boundaries.
